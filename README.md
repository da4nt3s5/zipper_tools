## zipper_tools / zipper_tools

## Project Overview

`zipper_tools` is a tool-orchestration server called **zipper_tools**. It is a FastAPI-based REST API that:
1. Maintains a registry of external analysis tools sourced from GitHub repositories.
2. Accepts analysis jobs (a file upload or a URL) via its API.
3. Dispatches each job to all registered tools that support its input type.
4. Persists job state and results to the filesystem.

Primary use-case: security and analysis tooling (e.g., APK static analysis). The codebase is bilingual (Spanish user-facing messages, English code identifiers).

## Key Modules

### `server/app.py`
FastAPI app (`zipper_tools`). Three endpoints:

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/submit` | Submit a file (multipart) or URL (JSON body) for analysis |
| `POST` | `/tools/add` | Register a new tool from a GitHub repo URL |
| `GET` | `/jobs/{job_id}` | Retrieve job status and results |

`/submit` is synchronous: it blocks until all registered tools finish.

### `server/storage.py` — `JobStore`
Filesystem-backed job store. All data lives under `tools_runtime/work/{job_id}/`.

```
tools_runtime/work/{job_id}/
├── job.json         ← job metadata + results (JSON)
├── input/
│   └── sample.bin   ← uploaded file (file jobs only)
└── tools/
    └── {tool_id}/   ← per-tool output directory
```

Job lifecycle states: `queued` → `running` → `finished` (or `error`).

### `server/registry.py`
Thin wrapper around `tools_db.json`. The DB path is resolved relative to `__file__`, so the server can be started from any directory.

### `server/runner.py` — `run_job`
Iterates over all registered tools, filters by `accepts.kind`, and runs each tool's command template via `subprocess.run(cmd, shell=True)`.

Command template placeholders:
- `{file}` — absolute path to the uploaded file
- `{url}` — the submitted URL string
- `{outdir}` — per-tool output directory for this job
- `{jobdir}` — job root directory

### `server/tools_add.py`
Handles tool registration, both programmatically (called by `app.py`) and as a standalone CLI (`python server/tools_add.py <repo_url>`).

Steps:
1. Verify the repo is accessible (`git ls-remote`).
2. Clone the repo with `--depth 1` into `tools_storage/{tool_id}/repo/`.
3. Read `zipper_tools.yaml` from the cloned repo (or run an interactive wizard to create one).
4. Write the tool record into `tools_db.json`.

## Tool Manifest (`zipper_tools.yaml`)

Every tool repo must contain a `zipper_tools.yaml` at its root:

```yaml
accepts:
  kind: file          # "file" | "url" | "both"
  file_types:         # optional, only for kind=file or both
    - ext: [.apk, .exe]

entrypoint:
  cmd: "python run.py -f {file} -o {outdir}"   # shell command template

runtime:
  type: python        # "python" | "docker" | "binary" | "go"
```

## Development Workflow

### Prerequisites
- Python 3.10+, pip3, git

### Initial Setup
```bash
# From the repo root, run the installer (interactive — asks for port)
bash install.sh

# Then start the server
cd zipper_tools
./run_server.sh
```

The server listens on `0.0.0.0:3030` by default (set during install).

### Running Without the Installer
```bash
cd zipper_tools
source .venv/bin/activate
uvicorn server.app:app --host 0.0.0.0 --port 3030 --reload
```

### Adding a Tool via CLI
```bash
cd zipper_tools
source .venv/bin/activate
python server/tools_add.py https://github.com/example/mytool
```

### Adding a Tool via API
```bash
curl -X POST http://localhost:3030/tools/add \
  -H "Content-Type: application/json" \
  -d '{"repo_url": "https://github.com/example/mytool"}'
```

### Submitting a Job
```bash
# File job
curl -X POST http://localhost:3030/submit \
  -F "file=@/path/to/sample.apk"

# URL job
curl -X POST http://localhost:3030/submit \
  -H "Content-Type: application/json" \
  -d '{"type": "url", "url": "https://example.com/target"}'
```

### Checking a Job
```bash
curl http://localhost:3030/jobs/<job_id>
```

## Important Conventions

1. **Working directory**: The server can be started from any directory. All paths (`tools_db.json`, `tools_runtime/work`, `tools_storage`) are resolved relative to the location of each module file using `__file__`.

2. **Tool IDs**: 8-character hex prefix of a UUID4 (e.g., `2cef2cff`). Generated at registration time.

3. **Spanish UI strings**: Error messages, prompts, and print statements are in Spanish. Code identifiers (function names, variable names, keys) are in English.

4. **Job storage is ephemeral**: `tools_runtime/work/` is runtime state; do not rely on it persisting across server restarts in production.

5. **`tools_db.json` paths**: The `path` field in each tool record stores the absolute path from the machine where the tool was registered. These paths will be wrong on a different machine; update them after migrating.

## Known Limitations & Security Notes

- **No authentication**: All API endpoints are unauthenticated. Do not expose this service to untrusted networks.
- **Shell injection risk**: `runner.py` passes tool command strings through `shell=True`. Tool manifests must come from trusted sources.
- **Synchronous job execution**: `/submit` blocks until all tools complete. Long-running tools will time out HTTP clients.
- **Absolute paths in `tools_db.json`**: Tool records embed absolute filesystem paths that are machine-specific.
- **No test suite**: There are currently no unit or integration tests in this repository.
